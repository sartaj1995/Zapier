# Real-Time Data Systems Anomalies Alerts Engine

## Business Problem Statement
System errors, database drops, and data pipeline disruptions can bring corporate operations to a screeching halt. Relying on engineers or IT administrators to manually refresh status pages or monitor server logs results in severe response delays, often allowing bugs to cause cascading data corruption before they are even discovered. 

This project solves these operational blind spots by transforming passive system logging into an active, real-time alarm system. By bridging standard syndication feeds, a precise text-filtering logic layer, and immediate communication APIs, this workflow monitors any live XML system log 24/7. It screens incoming data for anomalies and drops a high-priority incident alert straight into a designated email inbox the moment a system failure occurs.

## System Architecture & Data Flow
The pipeline utilizes a multi-step automation architecture built in Zapier to pull live server updates, isolate high-risk bugs, and route notifications instantly.

* **Trigger (RSS by Zapier):** Monitors a targeted, system-generated RSS status feed URL (such as server error logs or website uptime XML endpoints). The Zap operates on a "New Item in Feed" polling cycle, waking up the instant a new status entry is generated by the monitored infrastructure.
* **Data Transformation & Logic Layer (Filter by Zapier):** Because a healthy server feed outputs hundreds of normal, routine operational logs every day, a filter step acts as an automated triage. It inspects incoming payloads and only permits the workflow to continue if specific anomaly indicators are present (e.g., when the title string text explicitly contains the word "error").
* **Incident Notification Layer (Gmail API):** Once an anomaly successfully passes the filter rules, Zapier maps the live error details directly into a clean notification template. It executes a "Send Email" action to broadcast the precise system error message, link, and timestamp to the engineering team's inbox for immediate mitigation.

## Implementation & Customization

To deploy this automation pattern for your own infrastructure, configure the variables within the steps as follows:

### 1. Feed Configuration (Step 1)
Replace the source feed properties with your own live monitoring endpoint:
* **Feed URL:** Provide your internal system status XML URL (e.g., `https://yourdomain.com/status/feed.xml`)
* **What triggers a new item:** Choose *Different GUID/URL* to ensure every unique event log is registered independently.

### 2. Condition Ruleset (Step 2)
Modify the filtering parameters based on your specific infrastructure text patterns:

```text
Filter Criteria:
Field: Title
Condition: (Text) Contains
Value: error
